# KMRL NexusAI — Business Continuity Plan (BCP)
## Manual Fallback · Degraded Mode · Offline Scheduling

**Document Owner**: Operations Manager  
**Last Updated**: 2025-06-01  
**Review Frequency**: Semi-annual  
**Test Frequency**: Quarterly

---

## Purpose

This BCP ensures Kochi Metro Rail Limited can maintain nightly train induction
planning operations even when the NexusAI platform is fully unavailable.
It defines manual fallback procedures that allow depot controllers and
operations teams to plan without AI assistance.

**Key Principle**: *The platform enhances operations; it never replaces human
judgment. Every decision made by the AI can be made manually. This document
is the manual.*

---

## Trigger Conditions for BCP Activation

Activate this BCP when:

| Condition | Activation Level |
|-----------|-----------------|
| Platform unavailable > 30 minutes during 21:00–23:00 IST window | Full BCP |
| Optimizer not responding; no plan generated by 21:30 | Partial BCP (manual planning only) |
| Database inaccessible; fleet status unknown | Full BCP |
| Cyber incident requiring platform isolation | Full BCP |
| Natural disaster / power failure at data center | Full BCP |
| Scheduled maintenance during planning window | Planned BCP (planned 48h in advance) |

## Section 1 — Manual Fleet Status Assessment

When NexusAI is unavailable, determine fleet status from:

### Primary Sources (in priority order)

1. **Physical depot inspection** — Depot Foreman walks the yard and records which bays are occupied and by which trainsets

2. **Last printed status sheet** — Keep a printed copy of yesterday's approved induction plan at the Depot Control Room

3. **IBM Maximo mobile app** — Open work orders visible offline in cached Maximo mobile client

4. **Maintenance logbook** — Physical A4 logbook at each maintenance bay, updated after every service

## Section 2 — Manual Induction Planning Rules

Apply these rules in order when the AI optimizer is unavailable.

### Rule 1: Hard Exclusions (MANDATORY)

Trainsets that MUST NOT enter revenue service:

```
□ Any trainset with expired fitness certificate
  → Check certificate binder in Depot Control Room
  → Certificates expire dates marked in RED in the binder

□ Any trainset with open CRITICAL job card
  → Check IBM Maximo (or paper job card rack)
  → Red-tagged job cards = critical; must be cleared before service

□ Any trainset with brake health < 60%
  → Check last brake test record in maintenance log

□ Any trainset physically tagged OUT OF SERVICE
  → Red physical tag on front of trainset in depot
```

### Rule 2: IBL Assignment

Assign to IBL if ANY of:
- Days since last IBL > 90 days (check IBL register)
- Brake health between 60–70% (schedule inspection)
- Maintenance supervisor flags for inspection

### Rule 3: Revenue Service Priority

Rank eligible trainsets by:

1. **First priority**: Trainsets with active branding contracts that are below SLA
   → Check branding contract folder (Depot Control Room drawer labelled "BRANDING")
   → If contract requires ≥ 40 hrs/week and current week total < 32 hrs → prioritize

2. **Second priority**: Trainsets with lowest mileage this week
   → Mileage log on whiteboard (updated daily by shift supervisor)
   → Assign trainsets furthest below fleet average

3. **Third priority**: Trainsets in most accessible bays (minimize shunting)
   → Prefer Row A (front rows) over Row B for quick deployment
   → Avoid moving a trainset that requires 3+ shunting operations

### Rule 4: Minimum Revenue Service Count

**MINIMUM**: 17 trainsets in revenue service (1 below AI target of 18)
**TARGET**: 18 trainsets in revenue service
**MAXIMUM STANDBY**: 4 trainsets (anything beyond = under-utilization)

If fewer than 17 eligible trainsets exist:
1. Contact Maintenance Supervisor immediately
2. Assess if any maintenance can be deferred 24 hours
3. If < 15 eligible → inform GM Operations

### Rule 5: Shunting Minimization

Manual shunting guidance:
```
To minimize shunting operations:
  1. Use trainsets already in Row A first (they face outward)
  2. If Row A trainset is excluded, use nearest Row B trainset
  3. IBL movements: plan as first movements (before 22:00)
     so IBL bay is free by 23:00
  4. Never plan >3 back-to-back single-track shunts
     (requires a "setback" movement which doubles time)
  5. Cleaning: schedule 2-3 trainsets per cleaning bay slot,
     not all at once (cleaning team handles 1 at a time)
```

## Section 4 — Degraded Mode Operations

When platform is partially available:

### Scenario: API available but optimizer failed

Action: Use API to fetch fleet status, but apply manual planning rules.
```
1. Export fleet data: GET /api/v1/fleet → save to clipboard
2. Apply Rules 1–5 manually
3. Enter plan directly via Fleet page status overrides
4. Log override reason: "BCP - optimizer unavailable"
```

### Scenario: Database available but frontend down

Action: Use API directly via curl or Postman (keep bookmarks).
```bash
# Get fleet status
curl -H "Authorization: Bearer $TOKEN" https://api.nexusai.kmrl.in/api/v1/fleet

# Override trainset status
curl -X PATCH \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"status":"revenue_service","reason":"Manual BCP override"}' \
  https://api.nexusai.kmrl.in/api/v1/fleet/TS-01/status
```

### Scenario: Full offline (no internet/intranet)

Action: Complete manual planning per Sections 1–3. Use Maximo mobile (cached).
After platform restoration, retroactively enter plan for audit trail.

## Section 5 — Platform Restoration Procedure

When NexusAI becomes available after BCP activation:

```
Step 1: Verify platform health
  □ Check https://nexusai.kmrl.in — platform loads
  □ API health: curl https://api.nexusai.kmrl.in/health
  □ Optimizer test: run a test optimization

Step 2: Reconcile manual plan with platform
  □ Open Fleet page
  □ For each trainset: verify current_status matches manual plan
  □ Update any discrepancies with reason "BCP reconciliation"
  □ All updates logged to audit_logs automatically

Step 3: Enter BCP event record
  □ Go to platform Settings → Incident Log (or use API)
  □ Record: start time, end time, reason, manual plan hash,
    number of trainsets affected, any deviations from AI guidance

Step 4: Post-event review (within 48 hours)
  □ Were any differences between manual plan and AI guidance significant?
  □ Did any manual decisions lead to worse outcomes (mileage, conflicts)?
  □ Update this BCP if any steps were unclear or missing

Step 5: Deactivate BCP
  □ Notify all stakeholders: "NexusAI restored. Normal operations resumed."
  □ Close BCP activation in Operations Logbook
```
